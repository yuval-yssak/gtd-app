# Sync Audit Findings — 2026-04-21

First real run: **6 pass · 2 fail · 9 intentional skip**. Both failures are real product bugs, not flakes. The failing scenarios are B1 (modify instance in GCal) and B4 (edit master title in GCal).

Diagnostics confirmed via extra logging and a 1.1s pre-patch sleep on B4 (the sleep did not rescue B4, which rules out the 5s echo window as the sole cause).

## Bug A — `link-routine` leaves `lastSyncedNotes` unset

**Location:** `api-server/src/routes/calendar.ts:607–644` (`POST /integrations/:id/link-routine/:routineId`).

**What:** After pushing a new recurring event to GCal, the routine is replaced with `{ ...routine, calendarEventId, calendarIntegrationId, updatedTs }` but `lastSyncedNotes` is never populated with the HTML description we just wrote to GCal.

**Consequence:** On the next sync:

1. `syncRoutineExceptions` calls `provider.getExceptions(..., masterContent = { title, description: routine.lastSyncedNotes ?? '' })`. With `lastSyncedNotes = undefined`, `masterContent.description === ''`.
2. Every expanded instance's real `description` (inherited from the master we just wrote — e.g. `<p>original</p>`) compares unequal to `''`.
3. `buildModifiedException` (`GoogleCalendarProvider.ts:400–420`) sets `descChanged = true` on every instance → returns ~13 bogus "modified" exceptions per year.
4. `resolveInboundNotes` on the master path sees `gcalDescription != lastSyncedNotes` and mistakes the unchanged master for an inbound change.

**Visible in the diagnostic run** (`src/tests-sync-audit/scenarios/B.gcal-originated.audit.ts` trace): `title=...B4 MasterEdit exceptionCount=13` on a routine that had zero real instance edits.

**Fix sketch:** in the link-routine route, after `createRecurringEvent` succeeds, read the created event back (or pass the HTML we just generated via `markdownToHtml`) and set `updatedRoutine.lastSyncedNotes = htmlWeJustPushed`.

## Bug B — `updateRoutineFromGCal` clobbers freshly-written `routineExceptions`

**Location:** `api-server/src/routes/calendar.ts:915–952` (`updateRoutineFromGCal`) invoked from `importRecurringEventAsRoutine` which runs *after* `syncRoutineExceptions` in the same sync cycle.

**What:** `syncSingleCalendar` flow (line 682–712):

1. `Promise.all(linkedRoutines.map((r) => syncRoutineExceptions(r, …)))` — writes `routineExceptions` via `routinesDAO.replaceById`.
2. `fetchEventsWithSyncToken` returns GCal events including the master.
3. `importCalendarEvents` → `importRecurringEventAsRoutine` → `updateRoutineFromGCal(existing, …)` where `existing` is a stale snapshot fetched in step 2's `routinesDAO.findArray` **before** step 1's write.
4. `replaceById(routineId, { ...existing, …updates })` — `existing.routineExceptions` is the pre-exception-sync value, so the just-written exceptions are dropped.

This only fires when `updateRoutineFromGCal` reaches `replaceById` — guarded by `structurallyNewer || notesUpdate`. With Bug A, `notesUpdate` is spuriously truthy for every synced routine after linking, so this race fires every time.

**Why B2/B3 still pass:** they read the exception immediately after the SAME sync call. The sync's `syncRoutineExceptions` runs first (writing exceptions) and THEN `importRecurringEventAsRoutine` runs with the stale snapshot. But for B2/B3 — these are run in a later test after B1/B4's routine's master has already been processed once, and subsequent syncs don't re-trigger the overwrite because `structurallyNewer` becomes false. B1 specifically is clobbered by its own sync.

**Fix sketch:** either (a) re-fetch `existing` inside `updateRoutineFromGCal` just before `replaceById` (same pattern used in `deactivateRoutineFromGCal` at line 978), or (b) use targeted `updateOne({$set: {…}})` instead of a full `replaceById` so exception list isn't touched.

## Bug C — `updatedTs` precision / clock mismatch drops legitimate master updates

**Location:** `api-server/src/routes/calendar.ts:922` in `updateRoutineFromGCal`:
```ts
const structurallyNewer = event.updated > existing.updatedTs;
```

**What:** `existing.updatedTs` is ISO8601 with millisecond precision (`2026-04-21T06:45:28.141Z`). GCal's `event.updated` is returned as seconds-precision (`2026-04-21T06:45:28.000Z`). If the GCal patch lands within the same wall-clock second as `link-routine`'s write, `event.updated < existing.updatedTs` → the structural update is skipped.

The test added a 1.1-second sleep before `patchMasterEvent` in B4 and the failure reproduced, so while this bug is real it was not the sole cause of today's B4 failure — Bug A's `notesUpdate` path still runs `replaceById` with stale fields for the title specifically because the `structurallyNewer` branch is the only one that copies `event.title` (lines 932–939). With `structurallyNewer = false`, `title` is never updated no matter what else happens.

**Fix sketch:** compare with second-level truncation, or compare `>=`, or decouple title updates from the `structurallyNewer` branch.

## Update — 2026-04-21 07:09 UTC

Applied fixes A, B, C and rediscovered a **fourth** bug while verifying B1:

### Bug D — `getExceptions` `timeMax` anchored to `since` instead of `now`

**Location:** `api-server/src/calendarProviders/GoogleCalendarProvider.ts:352`.

**What:** on the very first sync, `config.lastSyncedTs` is unset, so `since = dayjs(0).toISOString()` = `1970-01-01`. The query then uses `timeMax = dayjs(since).add(1, 'year')` = `1971-01-01`, returning zero events. Subsequent syncs have a non-zero `lastSyncedTs`, so the window becomes valid — which is why only the first test in each shared-account run (B1) hit this.

**Fix applied:** anchor `timeMax` to `dayjs()` (now + 1 year).
